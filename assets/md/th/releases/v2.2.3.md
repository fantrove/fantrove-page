**TL;DR** — แก้ต้นเหตุที่ประวัติการอัปเดตไม่ถูกบันทึกตอน deploy: สคริปต์ release เทียบเวอร์ชั่นใหม่กับตัวเองแทนที่จะเทียบกับเวอร์ชั่นก่อนหน้าจริงๆ และ CI workflow ยังตรวจสอบ path ไฟล์แบบเก่าที่ไม่มีอยู่แล้ว

## เกี่ยวกับการแก้ไขนี้

ระบบ deploy เสร็จสมบูรณ์ทุกครั้ง แต่ไม่เคยมีไฟล์ประวัติของเวอร์ชั่นก่อนหน้าถูกสร้างขึ้นเลย สาเหตุคือสคริปต์ release ตัดสินว่า "เวอร์ชั่นก่อนหน้า" คืออะไรโดยอ่าน current.md จาก git HEAD แต่พอถึงเวลาที่ CI รันจริง HEAD คือ commit ที่มีเวอร์ชั่นใหม่ไปแล้ว เพราะนักพัฒนา commit และ push การ bump เวอร์ชั่นไปก่อนที่ CI จะเริ่มทำงาน ทำให้สคริปต์เห็นว่า "เวอร์ชั่นก่อนหน้า" กับ "เวอร์ชั่นใหม่" เป็นค่าเดียวกันเสมอ จึงสรุปว่าไม่มีอะไรเปลี่ยนแปลงและข้ามการบันทึกประวัติไปทั้งหมด

นอกจากนี้ CI workflow ยังตรวจสอบและ commit โครงสร้างไฟล์ release แบบเก่าระดับ root ที่ถูกยกเลิกไปแล้วตั้งแต่ย้ายประวัติไปไว้ในโฟลเดอร์ releases ของแต่ละภาษา ทำให้ต่อให้ snapshot ถูกสร้างขึ้นถูกต้อง ก็จะไม่ถูกตรวจพบหรือ commit อยู่ดี

### Fixed

- **สคริปต์ release หาเวอร์ชั่นก่อนหน้าได้ถูกต้องแล้ว**
  แทนที่จะสมมติว่า HEAD เป็น commit เก่าเสมอ สคริปต์จะตรวจก่อนว่าเวอร์ชั่นใน HEAD ตรงกับเวอร์ชั่นใหม่หรือไม่ ถ้าตรงกัน (กรณี CI รันหลัง push) จะย้อนดูประวัติ git ของไฟล์นั้นเพื่อหา commit ที่มีเวอร์ชั่นก่อนหน้าจริงๆ วิธีนี้ถูกต้องทั้งตอนรันก่อน commit ในเครื่อง และตอนรันหลัง push ใน CI

- **ปรับ CI workflow ให้ตรงกับโครงสร้าง releases ต่อภาษา**
  ขั้นตอนตรวจสอบ build และการ auto-commit ตอนนี้ตรวจสอบและ stage `assets/md/en/releases/` และ `assets/md/th/releases/` ให้ตรงกับสิ่งที่สคริปต์ release สร้างขึ้นจริง ทำให้ไฟล์ประวัติถูกตรวจสอบและ commit ได้ถูกต้องทุกครั้งที่ deploy
  ระบบยังคงสร้างไฟล์ประวัติในโฟลเดอร์ releases อัตโนมัติเมื่อมีเวอร์ชั่นใหม่ นักพัฒนาไม่ต้องสร้างหรือจัดการไฟล์ในโฟลเดอร์ releases เอง แค่แก้ไข current.md เท่านั้น

### สิ่งที่คุณจะสัมผัสได้

- โครงสร้างไฟล์ชัดเจน — current.md แยกจาก history
- นักพัฒนาทำงานง่ายขึ้น — แก้ไขแค่ current.md เท่านั้น
- ประวัติยังคงถูกบันทึกอัตโนมัติเหมือนเดิม
